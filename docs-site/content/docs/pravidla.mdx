## Uživatelé a role

**Interní** (`kind=internal`):

- `Admin`, `DeliveryManager`, `L1`, `L2`, `L3`

**Partner** (`kind=partner`):

- `responsible` — může vytvářet partner tickety, spravovat účastníky u system ticketů
- `technical` — nemůže vytvářet tickety; může být účastník

Neaktivní uživatelé se nemohou přihlásit ani pracovat s tickety.

## Typy ticketů

Definované v `backend/ticketmaster/models/constants.py`, např. `Question`, `Problem`, `Change Request`, `Security Issue`, …

Priorita: `Low`, `Normal`, `High`, `Critical`.

## Workflow stavů

Povolené přechody jsou v `WORKFLOW_TRANSITIONS` (stejný soubor). Zkráceně:

- `New` → `Assigned`, `Need more info`, `Rejected`, …
- `In progress` → `Resolved`, …
- `Resolved` / `Rejected` / … → `Closed`

API vrací `available_transitions` v detailu ticketu podle aktuálního stavu a role.

## Izolace partnerů

Partner vidí jen tickety svého `partner_id`. Testy v `backend/tests/test_business_rules.py` to pokrývají (`test_partner_isolation`).

## Důležitá oprávnění

| Akce | Kdo |
| --- | --- |
| Vytvořit partner ticket | `responsible` partner |
| Vytvořit interní ticket | interní uživatel |
| Ticket jménem partnera | Admin nebo Delivery Manager |
| Uzavřít ticket | Admin nebo Delivery Manager |
| Interní poznámka | interní, kdo ticket vidí |
| Komentář | účastníci komunikace ticketu |
| Editace/smazání komentářů | zakázáno (service vrací PermissionDenied) |

## System vs partner tickety

- **Partner ticket** — vlastník je partner user, vazba na klienta.
- **System ticket** — vytváří Admin/DM nebo partner API (responsible); přiřazení týmu L1/L2/L3.

## Meta API

Seznam enum hodnot pro UI: `GET /api/meta` (typy, priority, stavy, týmy).
