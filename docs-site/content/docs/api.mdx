Backend je **FastAPI** s prefixem **`/api`**. OpenAPI specifikace je generovaná automaticky z kódu (jediný zdroj pravdy) a publikovaná na **`/api/openapi.json`**.

Interaktivní reference (Scalar) je na této stránce. V produkci jsou **`/docs/*`** i **`/api/openapi.json`** dostupné jen z interní sítě (RFC1918 + loopback) — viz [Provoz](/docs/provoz).

## Přehled

| Položka | Hodnota |
| --- | --- |
| Base URL (dev) | `http://localhost:3006` |
| API prefix | `/api` |
| OpenAPI JSON | `/api/openapi.json` |
| Formát těla | `application/json` (kromě uploadu příloh) |
| Autentizace | `Authorization: Bearer <JWT>` |

Veřejné endpointy bez tokenu: `GET /api/health`, `GET /api/ready`, `GET /api/meta`.

## Autentizace

### Tok přihlášení (partner)

1. Klient pošle `POST /api/auth/login` s e-mailem a heslem.
2. API vrátí `{ "token": "<JWT>", "user": { ... } }`.
3. Další požadavky posílají hlavičku `Authorization: Bearer <JWT>`.
4. Profil aktuálního uživatele: `GET /api/auth/me`.

### Tok přihlášení (interní, dev)

V dev prostředí lze pro interní uživatele (`kind=internal`) použít `POST /api/auth/dev-sso` s e-mailem — uživatel musí existovat v DB.

### Aktivace pozvánky

Pozvaný uživatel volá `POST /api/auth/activate` s `token` z invite a novým `password`. Odpověď obsahuje JWT stejně jako login.

### Rate limiting loginu

Login (`/api/auth/login`, `/api/auth/dev-sso`) je rate-limited podle IP + e-mailu (`LOGIN_RATE_LIMIT_ATTEMPTS`, `LOGIN_RATE_LIMIT_WINDOW_SECONDS`).

| Situace | HTTP |
| --- | --- |
| Špatné heslo / neoprávněný uživatel | `403` |
| Příliš mnoho pokusů | `429` |

### Příklad: login a profil

```bash
BASE=http://localhost:3006

# Přihlášení partnera
curl -sS -X POST "$BASE/api/auth/login" \
  -H 'Content-Type: application/json' \
  -d '{"email":"partner@example.test","password":"ChangeMe123!"}'

# Uložte token z odpovědi
TOKEN="<jwt-z-loginu>"

# Profil přihlášeného uživatele
curl -sS "$BASE/api/auth/me" \
  -H "Authorization: Bearer $TOKEN"
```

## Konvence požadavků a odpovědí

### Hlavičky

| Hlavička | Směr | Popis |
| --- | --- | --- |
| `Authorization` | Request | `Bearer <JWT>` u chráněných endpointů |
| `Content-Type` | Request | `application/json` (kromě multipart uploadu) |
| `X-Request-ID` | Response | UUID pro korelaci s logy API |

### Chybové odpovědi

Aplikační chyby vrací JSON:

```json
{ "detail": "Popis chyby" }
```

| HTTP | Význam | Typické příčiny |
| --- | --- | --- |
| `400` | Validační / business chyba | Neplatná data, unsupported attachment |
| `403` | Zakázáno | Chybí token, špatná role, neplatný login |
| `404` | Nenalezeno | Ticket, komentář, příloha neexistuje |
| `409` | Konflikt | Business rule conflict |
| `422` | Nevalidní tělo požadavku | Pydantic validace (FastAPI) |
| `429` | Rate limit | Příliš mnoho login pokusů |

### Stránkování a filtry (`GET /api/tickets`)

Query parametry:

| Parametr | Typ | Popis |
| --- | --- | --- |
| `search` | string | Fulltext v titulku / popisu |
| `status` | string | Filtr stavu |
| `priority` | string | Filtr priority |
| `type` | string | Filtr typu ticketu |
| `resolver_team` | string | Filtr týmu řešitelů |
| `partner_id` | string | Filtr partnera (interní) |
| `internal` | bool | `true` / `false` — interní vs. partnerské tickety |
| `limit` | int | Počet záznamů (default z `TICKET_PAGE_DEFAULT_LIMIT`, max `TICKET_PAGE_MAX_LIMIT`) |
| `offset` | int | Offset pro stránkování (default `0`) |

Odpověď:

```json
{
  "items": [ /* ticket objekty */ ],
  "total": 123,
  "limit": 50,
  "offset": 0
}
```

Seznam je seřazen podle `created_at` sestupně (nejnovější první). Řazení jiným polem API nepodporuje.

### Číselníky

`GET /api/meta` vrací platné hodnoty pro `ticket_types`, `priorities`, `statuses`, `resolver_teams` — použijte je při vytváření ticketů.

## Praktické curl příklady

Následující příklady předpokládají běžící stack na `http://localhost:3006` a platný JWT v proměnné `TOKEN`.

### Meta a health

```bash
BASE=http://localhost:3006

curl -sS "$BASE/api/health"
curl -sS "$BASE/api/ready"
curl -sS "$BASE/api/meta"
```

### Seznam ticketů

```bash
curl -sS "$BASE/api/tickets?status=open&limit=10&offset=0" \
  -H "Authorization: Bearer $TOKEN"
```

### Detail ticketu

```bash
TICKET_ID="<uuid-ticketu>"

curl -sS "$BASE/api/tickets/$TICKET_ID" \
  -H "Authorization: Bearer $TOKEN"
```

Odpověď obsahuje detail ticketu včetně `available_transitions` — seznam stavů, do kterých může aktuální uživatel ticket převést.

### Vytvoření ticketu (partner)

Nejdříve získejte platné hodnoty z `/api/meta` a volitelně `client_id` z `/api/clients`.

```bash
curl -sS -X POST "$BASE/api/tickets" \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{
    "type": "incident",
    "priority": "medium",
    "title": "Problém s přihlášením",
    "description": "Uživatel se nemůže přihlásit od 9:00.",
    "client_id": null,
    "participant_ids": []
  }'
```

### Komentář k ticketu

```bash
curl -sS -X POST "$BASE/api/tickets/$TICKET_ID/comments" \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"body":"Doplňující informace od partnera."}'
```

### Seznam komentářů

```bash
curl -sS "$BASE/api/tickets/$TICKET_ID/comments" \
  -H "Authorization: Bearer $TOKEN"
```

### Upload přílohy

```bash
curl -sS -X POST "$BASE/api/tickets/$TICKET_ID/attachments" \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@/path/to/logfile.txt"
```

Volitelně lze připojit ke komentáři: `-F "comment_id=<uuid-komentare>"`.

### Stažení přílohy

```bash
ATTACHMENT_ID="<uuid-prilohy>"

curl -sS -L "$BASE/api/attachments/$ATTACHMENT_ID/download" \
  -H "Authorization: Bearer $TOKEN" \
  -o downloaded-file.txt
```

### Změna stavu ticketu

```bash
curl -sS -X POST "$BASE/api/tickets/$TICKET_ID/transition" \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"status":"in_progress"}'
```

Používejte pouze stavy z `available_transitions` v detailu ticketu.

## Poznámky k integraci

- **OpenAPI vs. tato stránka:** Scalar zobrazuje kompletní specifikaci včetně schémat request/response. Tato stránka doplňuje provozní kontext, příklady a pravidla oprávnění.
- **Swagger UI na API kontejneru** je vypnuté (`docs_url=None`) — dokumentace je centralizovaná na `/docs/api`.
- **Partner API** (`/api/partner-api/...`) je určené pro automatizované vytváření system ticketů; běžný partnerský workflow používá `/api/tickets`.
- Každá odpověď obsahuje `X-Request-ID` — při hlášení problému uveďte tuto hodnotu spolu s časem požadavku.
